Hybrid Retrieval System

A technical guide detailing the architecture, implementation choices, and code functionality behind our production-ready RAG retrieval engine.

🔍 What is Hybrid Retrieval?

Modern Generative AI pipelines rely on context retrieval (RAG) to provide an LLM with relevant knowledge. Finding that context usually relies on one of two methods:

Dense Retrieval (FAISS)

Uses AI vector embeddings to understand the semantic meaning of a query. Excellent for conceptual questions, but frequently misses exact names, IDs, or specific jargon.

Sparse Retrieval (BM25)

Uses statistical keyword matching to find exact words. Highly precise for names or technical terms, but completely blind to synonyms or thematic meaning.

The Solution: This project combines both approaches. Dense retrieval captures the semantic intent, while Sparse retrieval anchors the exact phrasing. We merge these using an industry-standard algorithm called Reciprocal Rank Fusion (RRF).

Data Ingestion Layer (main.py)

  • PDF Loader: We abandoned basic loaders in favor of fitz (PyMuPDF). Mupdf correctly extracts raw text from files like 'The Bhagavad Gita' avoiding encoding bugs.
  • Filter Guardrails: Before indexing, text runs through an is_valid_chunk() filter. This completely strips out headers, page numbers, trailing spaces, and table-of-content garbage that would otherwise severely corrupt the BM25 lexical space.
  • LangChain Chunking: The RecursiveCharacterTextSplitter chops the clean text into 400-character segments with 80-character overlap to retain context across boundaries.
  • Unique ID Tagging: Critically, every chunk generated is assigned a unique integer doc_id in its metadata. This guarantees that when the Dense and Sparse scores merge, we map scores natively to the exact block of text mathematically, regardless of content duplication.

⚙️ The Core Fusion Engine

The entire hybrid intelligence sits inside hybrid_retriever.py.

1. Intelligent Tokenization

Standard BM25 splits on spaces, which easily fails due to punctuation (like commas or quotes connecting to names). We implemented a regex tokenization function that cleans input by enforcing lowercase and strictly substituting non-alphanumeric characters with spaces to guarantee exact keyword matches.

2. Candidate Pooling

Instead of requesting top 3 from Dense and top 3 from Sparse, we pull top 20 candidates initially from each index. This prevents valid chunks that ranked slightly lower in one vector space from getting prematurely deleted before the math merges them.

3. Reciprocal Rank Fusion (RRF) Math

We abandoned naive min-max scaling because BM25 outputs unbounded statistical scores (which broke the normalization). Instead, we map results to ranks. The fusion formula takes the rank placement of a document out of each system and converts it to a standard fractional weight:

score = 1.0 / (60 + rank)

By applying RRF to both Dense and Sparse results, we acquire perfectly bounded [0, 1] numbers. We then blend these using tunable alpha (dense weight) and beta (sparse weight) multipliers to get the ultimate final score for LLM context submission.

🚀 Generation & Verification

After merging the indexes, main.py tests the engine. We validate using three highly contextual prompts aligned with the English commentary nature of the dataset:

  • Keyword Test: Querying for explicit speaker details forces the BM25Okapi sparse index to do the heavy lifting.
  • Semantic Test: Asking "What does Krishna teach about duty?" forces the HuggingFaceEmbeddings to grasp thematic meaning across the chapters.

The top 3 combined contexts from the fusion engine are formatted into a prompt and submitted to ChatGroq (llama-3.1-8b-instant), mapping lightning-fast LLM generation directly against precision-isolated context.
