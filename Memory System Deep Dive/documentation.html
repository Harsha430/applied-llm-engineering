Memory System Deep Dive

An encyclopedic, A-Z technical manual detailing every file, every library, every parameter, and every Custom Function and Class execution flow in this stateful LangChain AI project.

1. The Project Architecture

Before diving into individual lines of code, you must understand the "blueprint" of the repository. This project is built using a decoupled architecture: the User Interface (the CLI) is completely separated from the AI Logic (the Summarizer).

Directory Structure

Root Directory/
├── main.py
├── .env
├── requirements.txt
└── memory/
    ├── __init__.py
    ├── summarizer.py
    └── test_memory.py

How Data Flows

1. The User launches main.py and enters a session_id (e.g., "harsha_01").

2. The User types a message. main.py captures it.

3. main.py sends the message AND the session_id to summarizer.py.

4. summarizer.py intercepts it, looks up "harsha_01"'s past history, trims the history if it's too long, attaches the new message, and sends it to Groq.

5. Groq replies, the memory is updated, and the response is sent back to the User.

2. In-Depth Function & Method Execution Flow

This is exactly how every single custom function operates behind the scenes. We explain the exact mechanics of execution, classes used, and what parameters they require.

Function: get_session_history(session_id: str) -> InMemoryChatMessageHistory

Location: memory/summarizer.py

Core Role: This function acts as the Database Controller. It is called automatically by the RunnableWithMessageHistory wrapper every time a message is sent or received.

Execution Mechanics:

  • Step 1: It receives the session_id string (e.g., `"user1"`).
  • Step 2: It checks our globally defined dictionary variable called store. It asks: "Does 'user1' already exist inside store?"
  • Step 3: If 'user1' is NOT found, it instantiates a brand new InMemoryChatMessageHistory() Class Object from LangChain Core, and assigns it to store["user1"]. This Class Object has native methods to internally save `HumanMessage` and `AIMessage` objects inside a list called `.messages`.
  • Step 4: It returns that specific Class Object so the LCEL chain can read from it and write to it.

Function: get_token_count(messages: List[BaseMessage]) -> int

Location: memory/summarizer.py

Core Role: Acts as the mathematical Token Calculator for the Trimmer.

Execution Mechanics:

  • Step 1: LangChain's trim_messages function automatically passes an array of BaseMessage objects to this function before every API call.
  • Step 2: This function creates a generator comprehension: (len(m.content) // 4 for m in messages). It loops over every single message object in the history.
  • Step 3: For each message, it extracts the raw string using m.content, counts the physical string characters using len(), and uses integer division // 4 to approximate the token count (Standard LLM rule of thumb: 1 token = 4 characters).
  • Step 4: It wraps this inside python's built-in sum() function to return one final integer representing the total token size of the entire conversation.

Function: chat_with_memory(session_id: str, human_input: str) -> str

Location: memory/summarizer.py

Core Role: The ONLY public API exposed to the frontend (main.py). This triggers an LLM inference cycle.

Execution Mechanics:

  • Step 1: Takes the raw string typed by the user (human_input) and the active Session ID string (session_id).
  • Step 2: Wraps the raw string inside Langchain's specific data-type Class called HumanMessage(content=human_input). This is mandatory because LangChain chains only understand internal Message objects, not raw strings.
  • Step 3: Triggers the execution chain by calling memory_chain.invoke().
  • Step 4: It passes the session_id deep into the LangChain wrapper via the strict parameter format: config={"configurable": {"session_id": session_id}}. This secret "configurable" dictionary tells the wrapper which history dictionary to pull!
  • Step 5: The pipeline finishes, LLM responds with an AIMessage class, and this function extracts the raw string out of it using response.content and returns it to `main.py`.

Function: get_session_messages(session_id: str) -> List[BaseMessage]

Location: memory/summarizer.py

Core Role: A utility function allowing the user to view their un-trimmed, full chat log via the terminal.

Execution Mechanics:

  • Step 1: Takes the current session_id string.
  • Step 2: Calls the Database Controller get_session_history(session_id) we reviewed earlier, which pulls the InMemoryChatMessageHistory Class Object.
  • Step 3: Directly accesses the public `.messages` attribute inside that Object class, which points to the raw Python list `[...]` containing every message object ever sent or received in that session. Returns the list.

Function: main()

Location: main.py

Core Role: Handles the infinite while-loop for Terminal User I/O and graceful exits.

Execution Mechanics:

  • Step 1: Sets up the application GUI (CLI outputs, welcome message) using python's basic print(). Uses Python's native input() to freeze the thread and wait for a user to type a session_id. Strips whitespace with .strip().
  • Step 2: Enters an infinite while True: loop, essentially running forever.
  • Step 3: Encapsulates the next input() prompt inside a try / except (KeyboardInterrupt, EOFError) block. If the user hits CTRL+C on their keyboard, Python's runtime violently throws a KeyboardInterrupt exception. This block catches the exception, prints "Exiting. Goodbye!", and cleanly executes a break command to shatter the while loop unharmed.
  • Step 4: Logical Routing. Checks if the user typed command words: If quit/exit -> breaks loop. If switch -> asks for new session_id and triggers continue to restart the loop early. If history -> invokes get_session_messages, loops over the arrays, and prints them out.
  • Step 5: If it's a normal chat message, prints "AI is thinking..." and invokes the external chat_with_memory function from `summarizer.py`, passing the session_id and user_input block. Prints the return variable.

3. The Ecosystem (Which Package Does What)

We did not reinvent the wheel. We utilized specific open-source libraries to construct this architecture.

Package Name Underlying Role in this Project
langchain The orchestrator. It provides the pipeline architecture (chains) that allows us to stream data from a memory bank, into a prompt, and then into an LLM sequentially.
langchain-core Provides the fundamental raw materials: SystemMessage, HumanMessage, RunnableWithMessageHistory, and trim_messages.
langchain-groq A highly specific API connector. Groq uses "LPUs" (Language Processing Units) which function differently than standard GPUs. This package translates standard LangChain commands into high-speed Groq API syntax.
python-dotenv Provides the load_dotenv() function. It seeks out the .env file in the directory, parses the string GROQ_API_KEY=..., and artificially injects it into os.environ.

4. Parameter & Variable Deep Dive (`summarizer.py`)

This is the surgical breakdown of the exact variables initialized in the code and what they mean to the LLM backend.

A. LLM Initialization

llm = ChatGroq(model="llama-3.1-8b-instant")

Why model="llama-3.1-8b-instant"? Llama 3.1 is Meta's state-of-the-art open-source model. The 8B (8 billion parameter) version is light enough to execute instantaneously on Groq's hardware while being smart enough to hold deep contextual conversations. It is the gold standard for real-time memory chatting.

B. The Master Trimmer (`trim_messages`)

Every LLM crashes if you send it too much text. To prevent this, we built a filter that dynamically deletes old messages just before the LLM crashes. Here are the exact parameters used and WHY.

Parameter Value Used The Massive Impact (Why)
max_tokens 2000 The Ceiling. This allows the AI to remember the last ~8,000 characters of the conversation (roughly 3 pages of text). If we used a tiny number (like 60), the AI would suffer from instant amnesia. By setting it to 2000, we guarantee a deep, long-term memory without ever breaching the LLM's physical context limits.
strategy "last" The Preservation Algorithm. When the 2000 limit is breached, what gets deleted? "last" tells LangChain to perfectly preserve the most recent messages, and permanently amputate the absolute oldest messages.
token_counter get_token_count The Ruler. LangChain doesn't know how to count tokens organically. We pass the custom get_token_count function we reviewed previously.
include_system True The Personality Lock. If history gets too long, older messages are deleted. But the very first message is the "System Message" (e.g. "You are a helpful AI"). If include_system was False, the AI would eventually delete its own instructions and forget who it is! True forces LangChain to permanently staple the System Prompt to the top of the memory.
allow_partial False The Anti-Corruption Rule. Imagine the cutoff limit hits right in the middle of a paragraph. If this was True, LangChain would chop the sentence in half ("The dog ran into the..."). Setting it to False forces it to gracefully drop the entire message node rather than feeding broken grammatical fragments to the LLM.
start_on "human" The Syntax Enforcer. API platforms enforce strict conversational syntax rules: an AI response cannot exist without a preceding Human question. If trimming accidentally resulted in the chat log starting with an AI response, the API would crash with a 400 Bad Request Error. start_on="human" guarantees the array ALWAYS begins legally.

C. The Execution Pipeline (LCEL)

chain = (
    RunnablePassthrough.assign(messages=itemgetter("messages") | trimmer)
    | prompt
    | llm
)

The Pipeline Flow:

1. RunnablePassthrough.assign intercepts the raw payload coming in {"messages": [...]}.

2. itemgetter("messages") plucks out the raw list of previous chat bubbles from that dictionary payload.

3. The | trimmer pipe forces the bloated chat history through our dynamic filtering function. The bloated array goes in, the safely truncated array comes out.

4. | prompt takes the output from the trimmer, formatting the truncated array alongside our System Prompt templates (ChatPromptTemplate.from_messages(...)).

5. | llm acts as the final destination. The fully prepared, token-perfect structure hits the `ChatGroq` invocation wrapper which queries the LPU servers.
